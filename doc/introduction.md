# PicSubToAnki 项目介绍

## 项目主要功能

`PicSubToAnki` 是一个把“字幕截图”快速制作成 Anki 卡片的桌面工具。

它的基本使用方式是：

1. 选择一个图片文件夹。
2. 程序读取其中的截图文件，并把图片文件名作为字幕句子来源。
3. 用户为每张图片输入要学习的目标单词。
4. 程序调用大模型分析该单词，生成：
   - 单词原形
   - 发音或音标
   - 释义
   - 高亮目标词的例句
   - 中文笔记说明
5. 程序再把图片、例句、发音链接等内容一起写入 Anki。

项目支持日语和英语两种模式。模式不是通过单独按钮切换，而是根据输入单词首字符来判断：

- 如果首字符编码值大于 `10000`，按日语模式处理
- 否则按英语模式处理

另外，它支持：

- 单张创建卡片
- 批量创建卡片
- 自动压缩图片后上传到 Anki 媒体库
- 如果同一个词已存在，则更新原有笔记，而不是只做重复新增

## 它如何与 Anki 通信

项目通过 `AnkiConnect` 插件与 Anki 通信。

通信方式是向本地地址 `http://localhost:8765` 发送 HTTP POST 请求，请求体为 JSON，格式包含：

- `action`
- `version`
- `params`

也就是说，这个项目不是直接操作 Anki 数据库，而是依赖 Anki 桌面端安装并启用 `AnkiConnect` 后，通过它提供的本地 API 完成操作。

代码中主要使用了这些 AnkiConnect 动作：

### 1. `storeMediaFile`

在创建卡片前，程序会先把原图压缩成 JPG：

- 最大宽度：`320`
- 最大高度：`240`
- 质量：`60`

然后将压缩后的图片上传到 Anki 媒体库。

### 2. `addNote`

当目标单词在 Anki 中不存在时，程序会创建一条新的笔记，并指定：

- 牌组名 `deckName`
- 模板名 `modelName`
- 各字段内容 `fields`

### 3. `findNotes` + `notesInfo`

程序会先按“单词字段”查询现有笔记，确认这个词是否已经存在。

查询逻辑大致是：

1. 按字段值查找
2. 再结合当前牌组查找
3. 如果还需要，再取出笔记详情逐条核对字段值

### 4. `updateNoteFields`

如果查询到该词已存在，程序不会直接重复新增，而是更新原笔记的几个字段：

- `例句`
- `笔记`
- `释义`

更新时会把新的例句和新图片继续追加进去。

## 它创建的 Anki 牌组格式和名称

这些配置来自项目根目录的 `config.ini`。

### 默认牌组名称

- 日语牌组：`日本語::ランダム::アニメ・マンガ・マスコミ`
- 英语牌组：`English Vocabulary::A English Daily`

项目启动时，`AnkiConnect` 默认先设置为日语牌组；实际创建时再根据输入单词判断使用哪一个牌组。

### 默认笔记模板名称

- `划词助手Antimoon模板`

这意味着程序创建的并不是某种内置固定卡型，而是把数据写入这个已有模板对应的字段中。

### 默认字段结构

配置中的字段映射如下：

- `单词`
- `音标`
- `释义`
- `笔记`
- `例句`
- `发音`

程序最终构造出的笔记内容大致是：

- `单词`：大模型返回的词条原形
- `音标`：读音或音标
- `释义`：词典式解释
- `笔记`：中文语境说明或学习备注
- `例句`：带有目标词高亮的例句，并在后面追加图片
- `发音`：`[sound:URL]` 格式的音频引用

其中发音链接来源为：

- 日语：LanguagePod101
- 英语：有道词典语音接口

## 补充说明

这个项目的实际流程可以概括为：

“读取截图文件 -> 取文件名作为字幕 -> 输入目标词 -> 调用大模型生成词条信息 -> 上传压缩图片到 Anki -> 新增或更新笔记”

因此它本质上是一个“字幕截图制卡工具”，其中：

- 大模型负责词条解析和内容生成
- AnkiConnect 负责写入 Anki
- `config.ini` 负责定义最终写入哪个牌组、哪个模板、哪些字段

## LLM 调用是如何配置的

这个项目里的大模型调用配置，主要由根目录下 `config.ini` 的 `[openai]` 段控制。

默认配置如下：

```ini
[openai]
api_key = sk-8d1b813e56b441b781136e9fc5de2eca
base_url = https://dashscope.aliyuncs.com/compatible-mode/v1
model_name = qwen-plus
```

这 3 个配置项的含义分别是：

- `api_key`：模型服务的 API 密钥
- `base_url`：模型接口的服务地址
- `model_name`：实际调用的模型名称

从这个配置可以看出，项目虽然使用的是 OpenAI 风格的 SDK，但并不限定只能连接 OpenAI 官方接口。只要后端服务兼容 OpenAI 接口格式，就可以通过修改 `base_url` 和 `model_name` 来接入。项目默认接的是阿里云 DashScope 的兼容接口，并默认使用 `qwen-plus`。

## LLM 配置是如何读入的

程序启动时，`main.py` 会读取 `config.ini`，把 `[openai]` 段中的这几个值取出来，组成 `self.openai_config`：

- `api_key`
- `base_url`
- `model_name`

随后程序并不会立刻初始化模型客户端，而是采用延迟初始化：

- 当用户第一次点击“创建卡片”
- 或第一次执行“批量添加”

才会创建：

```python
OpenAIExplanation(**self.openai_config)
```

这意味着模型连接只会在真正开始制卡时发生，而不是在界面启动时就立即建立。

## 实际调用用了什么库

项目在 `openai_utils.py` 中使用的是 `openai` Python SDK：

```python
from openai import OpenAI
```

客户端初始化代码等价于：

```python
OpenAI(
    api_key=self.api_key,
    base_url=self.base_url
)
```

这里的关键点是：

- `api_key` 决定认证
- `base_url` 决定连接到哪个模型服务商
- `model_name` 决定具体使用哪个模型

因此，这个项目本质上是在用 OpenAI SDK 作为统一入口，去调用一个“OpenAI 兼容接口”。

## LLM 请求是怎么发出去的

无论是单条解析还是批量解析，底层都调用：

`self.client.chat.completions.create(...)`

请求参数里最关键的是：

- `model=self.model_name`
- `messages=[...]`
- `temperature=0.1`
- `stream=False`

这说明它采用的是 Chat Completions 风格接口，而且：

- 温度较低，偏向稳定输出
- 不走流式返回，而是等待完整结果

这很适合当前这个项目，因为它需要模型尽量稳定地输出结构化 JSON，而不是更发散的自然语言回答。

## Prompt 是如何组织的

项目没有把 prompt 放进配置文件，而是直接写在 `openai_utils.py` 代码里。

目前分成四类：

- 日语单条：`construct_single_prompt_jp`
- 英语单条：`construct_single_prompt_en`
- 日语批量：`construct_batch_prompt_jp`
- 英语批量：`construct_batch_prompt_en`

每次调用时，prompt 的核心输入有两部分：

- `subtitle`：从图片文件名得到的字幕句子
- `key`：用户输入的目标单词

模型被要求返回结构化 JSON，主要字段包括：

- `单词`
- `音标`
- `意义`
- `例句`
- `笔记`

其中 prompt 还额外约束了输出风格，例如：

- 要把目标词在例句中用 `<b>...</b>` 包起来
- 要提供中文语境解释
- 日语和英语要分别按不同词典风格回答

## 模式和 LLM 调用之间的关系

项目没有单独做一个强制的语言切换配置，而是根据用户输入单词自动判断模式：

- 如果首字符编码值大于 `10000`，视为日语
- 否则视为英语

判断出模式后，程序会同步设置：

- `self.anki_connect.mode`
- `self.openai_client.mode`

这会直接影响后续 LLM 调用，因为 `OpenAIExplanation` 会根据 `mode` 选择对应的 prompt 构造函数。

所以这里的“模式”不仅决定写入哪个 Anki 牌组，也决定：

- 发送给模型的提示词内容
- 输出结果的语言风格
- 发音字段后续采用哪种音频来源

## 单条调用和批量调用

### 单条调用

单张创建卡片时，会调用：

`explain_single(subtitle, key)`

这个方法会：

1. 根据当前模式构造单条 prompt
2. 调用模型接口
3. 取得返回文本
4. 解析 JSON
5. 映射成内部统一字段：
   - `word`
   - `pronunciation`
   - `meaning`
   - `example`
   - `note`

### 批量调用

批量创建卡片时，会调用：

`explain_batch(subtitles, keys)`

它会把多组“字幕 + 单词”组合成一个批量 prompt，要求模型一次返回一个 JSON 数组。之后程序再把数组结果逐项拆开，对应回每一张图片。

这种做法的好处是：

- 减少请求次数
- 降低批量处理时的接口开销
- 让一批词条的输出风格更统一

## 模型返回结果如何解析

模型返回后，程序会先把文本取出来，再尝试解析 JSON。

解析逻辑包括：

1. 如果返回内容外层包着 ```json 代码块```，先去掉代码块标记
2. 调用 `json.loads(...)`
3. 如果失败，则认为这次大模型调用失败

解析成功后，中文字段会被映射为内部字段名：

- `单词` -> `word`
- `音标` -> `pronunciation`
- `意义` -> `meaning`
- `例句` -> `example`
- `笔记` -> `note`

可以看出，这个项目对模型输出格式的依赖比较强。如果模型没有严格返回合法 JSON，或者字段名不符合预期，后续制卡流程就会失败。

## 这种配置方式的特点

这里的 LLM 配置方式有几个明显特点：

- 简单：只需要在 `config.ini` 中填写 `api_key`、`base_url`、`model_name`
- 兼容：可切换到任何 OpenAI 兼容接口
- 固定：prompt 写在代码里，不是动态配置项
- 结构化：高度依赖模型输出 JSON

因此它最容易切换的是：

- 模型服务商
- 模型名称

而不太方便直接切换的是：

- prompt 模板
- 返回字段结构
- 不同任务类型的推理策略

## 一句话总结

这里的 LLM 调用，本质上是“用 OpenAI SDK 去访问一个 OpenAI 兼容模型接口，再通过固定 prompt 让模型输出结构化 JSON”，然后把这些 JSON 结果交给后续的 Anki 制卡流程继续处理。

## 项目设计要点

这个项目从实现上已经体现出一个很重要的设计方向：

- LLM 的调用配置需要显式配置
- Anki 的牌组和模板配置也需要显式配置

这里的“显式配置”意思是：

- 配置项应该集中放在 `config.ini` 这类明确的配置文件中
- 用户可以直接看到、修改和确认这些值
- 程序不应该把关键目标写死在代码里，再让用户靠阅读源码猜测当前行为

## 为什么 LLM 配置必须显式化

LLM 接入天然具有环境差异，不同用户可能会使用：

- 不同服务商
- 不同接口地址
- 不同模型名称
- 不同密钥

因此下面这些内容必须是明确可配置的：

- `api_key`
- `base_url`
- `model_name`

这样做的好处是：

- 更容易切换模型供应商
- 更容易在本地、公司内网、代理环境之间切换
- 出问题时更容易排查当前到底连的是哪个模型服务
- 避免把敏感信息硬编码进源码

从工程设计角度看，LLM 属于外部依赖能力，不应被实现成隐式常量，而应被设计成可替换、可检查、可迁移的配置项。

## 为什么 Anki 牌组配置也必须显式化

Anki 这一侧同样不能假设所有用户都使用同一套牌组结构。

不同用户的实际环境可能存在差异：

- 牌组名称不同
- 模板名称不同
- 字段名称不同
- 日语和英语牌组拆分方式不同

因此这些内容都应该显式配置：

- 日语牌组名 `jp_deck`
- 英语牌组名 `en_deck`
- 笔记模板名 `model_name`
- 各字段名映射：
  - `word_field`
  - `pronunciation_field`
  - `meaning_field`
  - `note_field`
  - `example_field`
  - `voice_field`

这样做的直接价值是：

- 用户不必为了适配自己的 Anki 模板去修改源码
- 项目可以适配已有卡组体系，而不是强迫用户迁移到固定模板
- 制卡逻辑和牌组结构之间的耦合度更低

## 这条设计原则对项目维护的意义

把 LLM 配置和 Anki 配置都做成显式配置后，项目会更容易维护：

- 新环境部署时，只需要调整配置文件
- 更换模型或更换牌组时，不需要改代码逻辑
- 不同用户可以共享同一套程序，但使用各自的配置
- 文档也更容易说明“哪些是行为，哪些是环境参数”

这其实是在把项目划分成两层：

- 稳定的业务逻辑层：读取图片、调用模型、写入 Anki
- 可变的环境配置层：模型服务、牌组名称、字段映射

这种分层是很合理的，因为真正经常变化的，往往不是流程本身，而是外部服务和用户自己的 Anki 结构。

## 对后续设计的启发

如果后面继续演进这个项目，那么一个明确的设计要求应该是：

- 所有会因用户环境不同而变化的内容，都优先考虑放进显式配置

至少包括：

- LLM 服务配置
- Anki 牌组配置
- Anki 模板名和字段映射

如果继续扩展，还可以考虑让这些内容进一步配置化：

- prompt 模板
- 语言模式判断规则
- 图片压缩参数
- 发音来源地址

## 一句话总结

这个项目的一个核心设计要点，就是把“模型接入方式”和“Anki 落卡目标”都视为外部环境配置，而不是硬编码的业务逻辑；只有这样，项目才能真正适配不同用户、不同模型平台和不同 Anki 牌组结构。
